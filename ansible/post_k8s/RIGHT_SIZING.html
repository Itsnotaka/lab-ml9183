Design specification

Q2.1 Infrastructure requirements

ChefMate course project · Chameleon Cloud (KVM@TACC) · Q2 DevOps/Platform submission

Cluster
3-node k3s
Node capacity
4 vCPU, 8 GiB RAM, 40 GiB disk
Storage
local-path PVCs for app and platform state
Configuration source
k8s/platform/values.yaml, k8s/staging/values.yaml

GPU request and limit are 0 / 0 for every service: no Kubernetes manifest in this deployment requests GPU resources.

Infrastructure requirements table

Per-service GPU, CPU, and memory requests and limits, with Chameleon evidence and the basis for right-sizing.

Service Namespace GPU CPU req/limit Mem req/limit Chameleon evidence and right-sizing basis
mealie chefmate-staging 0 / 0 250m / 1000m 512Mi / 1Gi Observed: kubectl top pods -A showed about 266Mi RAM during validation, and kubectl get pods -A -o wide showed the service healthy in Kubernetes. Rationale: request is about 2x observed steady-state memory and the 1 GiB limit preserves burst headroom for a small web application workload.
mealie-postgres chefmate-staging 0 / 0 250m / 1000m 512Mi / 1Gi Observed: mealie-postgres-pvc was Bound at 8Gi, and the service was healthy during live cluster validation. Rationale: this is a single-application Postgres instance with modest Q2 load, so a small guaranteed reservation plus conservative limit is appropriate relative to the node size.
postgres chefmate-platform 0 / 0 250m / 1000m 512Mi / 1Gi Observed: postgres-pvc was Bound at 10Gi and the platform database was running as the MLflow backend. Rationale: the workload is a lightweight metadata database for MLflow rather than a high-throughput transactional database, so the selected reservation is intentionally modest.
minio chefmate-platform 0 / 0 250m / 1000m 512Mi / 1Gi Observed: minio-pvc was Bound at 20Gi, and the bucket initialization job completed successfully. Rationale: MinIO stores MLflow artifacts for a small team-scale project, so the request remains low while the limit provides headroom for startup and artifact transfers.
mlflow chefmate-platform 0 / 0 250m / 1000m 512Mi / 1Gi Observed: MLflow was deployed behind the platform service on port 8000 and backed by Postgres plus MinIO. Rationale: this is a light UI/API workload for milestone-scale experimentation, so a 250m / 512Mi request preserves cluster headroom while 1 CPU and 1 GiB limit covers interactive use.

Why these values are reasonable

  • 250m CPU request is about 6.25% of a 4 vCPU node.
  • 1000m CPU limit is about 25% of a 4 vCPU node.
  • 512Mi memory request is about 6.25% of an 8 GiB node.
  • 1Gi memory limit is about 12.5% of an 8 GiB node.
  • This leaves substantial headroom for Kubernetes system components, storage provisioning, and temporary workload spikes while still reserving enough resources for stable service operation.

Chameleon evidence used

  • kubectl get nodes -o wide to confirm cluster size, node readiness, and host capacity context.
  • kubectl get pods -A -o wide to confirm service placement and healthy runtime state.
  • kubectl get pvc -A to confirm durable storage claims were bound for platform and application services.
  • kubectl top nodes and kubectl top pods -A to capture live utilization where metrics were available.

Metrics were incomplete for some worker-hosted workloads at capture time, so the final sizing decision combines direct measurements that were available, healthy pod runtime behavior, PVC-backed storage status, and the known leased node capacity.
