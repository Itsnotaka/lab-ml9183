Infrastructure requirements and right-sizing (Q2.1)

ChefMate course project · Chameleon Cloud (KVM@TACC) · Q2 DevOps/Platform

Cluster: 3-node k3s · per node 4 vCPU, 8 GiB RAM, 40 GiB disk · local-path PVCs · k8s/platform/values.yaml, k8s/staging/values.yaml

Abstract. Per-service CPU and memory requests and limits are set from live kubectl observation on Chameleon and conservative headroom for a small multi-tenant node layout. No workload requests GPUs (0 / 0 everywhere).

1. Per-service table

Requests, limits, and the evidence and rationale used for sizing.

Service Namespace GPU CPU req/limit Mem req/limit Evidence and basis
mealie chefmate-staging 0 / 0 250m / 1000m 512Mi / 1Gi Observed: kubectl top pods -A ~266Mi RAM; pods healthy. Rationale: request ~2× steady memory; 1 GiB limit for burst on a small web app.
mealie-postgres chefmate-staging 0 / 0 250m / 1000m 512Mi / 1Gi Observed: mealie-postgres-pvc Bound at 8Gi; healthy. Rationale: single-app Postgres at Q2 load; modest reservation with conservative limit vs. node size.
postgres chefmate-platform 0 / 0 250m / 1000m 512Mi / 1Gi Observed: postgres-pvc Bound at 10Gi; MLflow metadata DB. Rationale: light metadata workload, not high-throughput OLTP.
minio chefmate-platform 0 / 0 250m / 1000m 512Mi / 1Gi Observed: minio-pvc Bound at 20Gi; init job completed. Rationale: team-scale artifact store; low request, higher limit for startup and transfers.
mlflow chefmate-platform 0 / 0 250m / 1000m 512Mi / 1Gi Observed: exposed on port 8000 with Postgres and MinIO. Rationale: light UI/API for milestone use; 250m / 512Mi request preserves headroom; 1 CPU / 1 GiB limit for interactive use.

2. Share of node capacity

  • 250m CPU request ≈ 6.25% of a 4 vCPU node.
  • 1000m CPU limit ≈ 25%.
  • 512Mi memory request ≈ 6.25% of 8 GiB.
  • 1Gi memory limit ≈ 12.5%.
  • Leaves room for kubelet, storage, and spikes while keeping services schedulable.

3. Evidence commands

  • kubectl get nodes -o wide — cluster size and readiness
  • kubectl get pods -A -o wide — placement and health
  • kubectl get pvc -A — bound claims
  • kubectl top nodes, kubectl top pods -A — utilization where available

Metrics were incomplete for some workers at capture; sizing combines available measurements, healthy runtime, PVC status, and known node lease.
